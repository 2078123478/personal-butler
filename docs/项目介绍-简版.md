# Vigil — BNB 生态生活助手与 Agent 执行系统

> Vigil 把官方 skill、执行链路、主动触达与链上可信通信组合成一个面向 BNB 生态的生活助手。当前仓库公开的重点，是展示系统主路径、长期运行约束与可复验证据；这不等于终端用户需要自己处理底层运行时配置。

---

## 项目定位

Vigil 不是单点工具，而是一个持续运行的系统：

- 持续感知 Binance 生态信号，发现机会与风险
- 用 LLM 判断哪些信息值得打扰用户，约 87% 降噪
- 用克隆音色进行语音播报，并在更高紧急度下升级电话触达
- 在 `paper-first` 前提下运行执行策略，并由三层风控约束行为
- 让 Agent 通过 BNB 链上铭文建立身份、信任与加密通信

理解这份仓库时，更值得关注的是：**系统闭环是否成立、约束是否清晰、证据是否可复验**。Vigil 当前运行在 OpenClaw assistant runtime 之上，终端用户的实际入口是 Telegram / 语音 / 电话，多通道接入与会话编排由运行时承接。

### 当前已落地能力（非路线图）

以下能力在当前仓库中已有对应代码、测试或可运行验证路径：

- Signal Radar 实时轮询 Binance 公告 + Square，输出 `NormalizedSignal`
- LLM Signal Triage 批量审核 + 规则引擎降级
- Contact Policy 6 级注意力阶梯 + quiet hours + 频率限制
- CosyVoice 克隆音色语音合成 + Telegram 语音投递
- Twilio 电话触达（紧急升级路径）
- Telegram Inline Keyboard 回调闭环
- 套利引擎：六维成本模型 + 三层风控 + paper/live 自动切换
- Agent-Comm：钱包身份 + EIP-712 签名名片 + E2E 加密 + 连接生命周期
- 首批官方 Skills 适配器已接入主路径
- API 路由可直接调用验证

验证方式：`npm run demo:judge` 或逐个运行 `demo:living-assistant` / `demo:discovery`。

## 四大核心模块

### 1. 🔗 Agent-Comm — 链上可信通信层

Agent-Comm 的价值不在“更花哨的传输”，而在于把身份、信任与消息路径绑定到链上可验证对象。

![Agent-Comm 名片卡片 — 真实钱包身份](assets/agent-comm-card-real.jpg)

- **区块链即消息总线**：消息可通过交易 calldata 承载，减少中心化基础设施依赖
- **EIP-712 身份凭证体系**：ContactCard / TransportBinding / RevocationNotice 三类签名工件
- **端到端加密**：secp256k1-ECDH + AES-256-GCM，支持密钥轮换与前向安全
- **连接生命周期**：发现 → 邀请 → 接受/拒绝 → 确认 → 通信 → 撤销

### 2. 💰 执行引擎 — 从发现到执行的闭环

执行层关注的是**长期可运行性**，不是单次瞬时表现。

![套利引擎 PnL Performance](assets/pnl-performance.png)

- **六维成本模型**：手续费、滑点、延迟、MEV、Gas、尾部风险
- **风险调整模拟器**：只有风险调整后的净边际满足条件才进入执行
- **三层风控**：准入门控 → 熔断器 → 动态阈值
- **自动模式切换**：默认 `paper`，满足条件再进入 `live`，异常时自动降级

### 3. 📡 Living Assistant — 主动感知与打扰决策

Living Assistant 解决的不是“有没有信号”，而是“哪些信号值得打扰人”。

- **Signal Radar**：实时轮询 Binance 公告与 Square，统一标准化为 `NormalizedSignal`
- **LLM Signal Triage**：`80 -> 8 notify / 12 digest / 60 skip`，约 87% 降噪
- **Contact Policy**：把信号映射到 `silent -> call_escalation` 的 6 级注意力阶梯
- **降级策略**：LLM 不可用时回退规则引擎；live 条件不足时保持 `paper-first`

### 4. 📞 多渠道触达 — 用合适强度联系用户

多渠道投递的目标不是“多发一种消息”，而是让系统能根据紧急度选择合适的联系强度。

- Telegram 文本 + Inline Keyboard
- CosyVoice 克隆音色语音播报
- Twilio 电话触达（更高紧急度）
- One-Breath Voice Brief（当前目标约束为 ≤15 秒、≤3 句话）

**当前仓库中的语音简报样例**：

> “老大，BNB 生态刚上新币 KAT，今晚 12 点 UTC 开交易，还带 Seed 高风险标签。这个币波动可能会很大，但咱们链上监控里还没布防对应合约。建议先把 KAT 在 BSC 上的资金流和池子首小时进出量拉起来盯一下。”

## Skills Hub 融合方式

当前已完成首批核心官方 Skills 融合，重点不是罗列覆盖率，而是把官方能力接进 Vigil 的主路径。官方技能覆盖/阶段/runtime/输出可见性详见 `official-skills-manifest.json`。

| 产品阶段 | 已接入官方能力 | 作用 |
|---|---|---|
| Signal | Binance Announcements / Binance Square | 生态信号输入 |
| Market | `binance/spot` | 执行引擎市场上下文 |
| Readiness | `binance/assets` | 执行前置检查 |
| Enrichment | `binance-web3/query-token-info` / `binance-web3/query-token-audit` | 决策上下文与安全审计 |

扩展方向已规划判断增强层（`trading-signal`、`query-address-info`）与结果分发层（`binance/square-post`）。

在套利审批 API 输出中，`skillAttribution` 会直接展示 `requiredSkillsUsed` / `enrichmentSkillsUsed` / `distributionSkillsUsed` / `skillSources`，便于评审快速核对接入来源。

适配器模式的意义，在于把 Skill 接入变成系统集成问题，而不是每次都重写一遍产品链路。

## 三个可复用生态贡献

| 层次 | 贡献 | 价值 |
|------|------|------|
| 信任层 | Agent-Comm 链上可信通信 | 给 Agent 身份、信任与消息路径一个可验证载体 |
| 判断层 | Contact Policy + 6 级注意力阶梯 | 让被动 Skill 变成主动判断系统 |

这些能力的重点，在于它们能否作为更大产品系统里的长期可复用模块存在。
